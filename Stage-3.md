# Stage 3 Plan

## Цель этапа 3

Сделать файловое хранение в JSON так, чтобы программа умела:

1. Сохранять текущее состояние коллекций `Experiment`, `Run`, `RunResult` в файл.
2. Загружать это состояние обратно из файла.
3. Проверять файл до замены текущих данных.
4. После загрузки продолжать работу без конфликтов `id` и без потери связей и временных меток.

Ключевой принцип: этап 3 надо строить поверх уже готовых этапов 1 и 2, не ломая их архитектуру.  
Домен остается ответственным за инварианты, сервисы за коллекции и `id`, файловый слой только за перенос данных между памятью и JSON.

## Что не стоит делать

Не надо сериализовать доменные объекты напрямую в JSON и потом пытаться десериализовать их обратно в домен.  
Этот подход слишком жестко связывает файловый формат с внутренним устройством доменных классов и ломается на восстановлении `createdAt`, `updatedAt`, `id` и инвариантов.

Текущие наработки stage 3 лучше считать черновиком и почти полностью переписать.

## Целевая архитектура

Нужны четыре слоя ответственности:

1. `storage/persistence/file`
   Отвечает за чтение и запись JSON-файла.

2. DTO-слой файла
   Отдельные классы для структуры JSON, например:
   - `DataSnapshot`
   - `ExperimentRecord`
   - `RunRecord`
   - `RunResultRecord`

3. `FileValidator`
   Проверяет структуру файла, обязательные поля, уникальность `id`, корректность ссылок и базовую корректность значений.

4. Restore/mapper слой
   Преобразует DTO из файла в доменные объекты и обратно.

Идея такая:

`services -> domain objects -> file DTO -> JSON`

и обратно:

`JSON -> file DTO -> validate -> restore domain -> load into services`

## Какой JSON должен быть на выходе

Один файл с тремя логическими таблицами:

```json
{
  "experiments": [
    {
      "id": 1,
      "name": "Exp A",
      "description": "desc",
      "ownerUsername": "user",
      "createdAt": "2026-04-19T10:00:00Z",
      "updatedAt": "2026-04-19T10:00:00Z"
    }
  ],
  "runs": [
    {
      "id": 1,
      "experimentId": 1,
      "name": "Run A",
      "operatorName": "operator",
      "createdAt": "2026-04-19T10:05:00Z",
      "updatedAt": "2026-04-19T10:05:00Z"
    }
  ],
  "runResults": [
    {
      "id": 1,
      "runId": 1,
      "param": "pH",
      "value": 7.0,
      "unit": "pH",
      "comment": "ok",
      "createdAt": "2026-04-19T10:10:00Z",
      "updatedAt": "2026-04-19T10:10:00Z"
    }
  ]
}
```

Важно:

- даты лучше хранить как ISO-8601 строки;
- имена секций должны быть стабильными;
- связи задаются через `experimentId` и `runId`.

## Какие минимальные изменения нужны вне stage 3

Этапы 1 и 2 не надо переписывать, но без небольших точечных изменений stage 3 нормально не сделать.

Нужно добавить restore-контракт:

1. Для `Experiment`
   Нужен способ создать объект не только как "новый сейчас", но и как "восстановленный из файла" с уже готовыми:
   - `id`
   - `createdAt`
   - `updatedAt`

2. Для `Run`
   То же самое.

3. Для `RunResult`
   То же самое, обязательно с `updatedAt`, которое сейчас у вас даже не сохраняется полностью.

Это можно сделать одним из двух способов:

- отдельные static factory методы вроде `restore(...)`;
- отдельные конструкторы для восстановления.

Лучше factory-методы `restore(...)`, чтобы явно разделить:

- создание нового объекта приложением;
- восстановление существующего объекта из файла.

## Что должны делать сервисы

Сервисы по-прежнему владеют коллекциями и `nextId`.

Им нужно уметь:

1. Загрузить уже восстановленные доменные объекты в коллекцию.
2. Пересчитать `nextId` как `maxId + 1`.
3. Заменять текущие данные только после полной успешной проверки и восстановления.

Если это уже частично есть, можно переиспользовать, но проверить контракт:

- сервис не должен генерировать новый `id` при restore;
- сервис не должен ломать порядок связей;
- замена данных должна быть атомарной на уровне сценария `load`.

## Что должен делать FileStorage

`FileStorage` должен заниматься только технической работой с файлом:

1. Прочитать JSON из пути.
2. Преобразовать JSON в `DataSnapshot`.
3. Записать `DataSnapshot` в JSON.

Он не должен:

- знать про бизнес-логику сервисов;
- сам восстанавливать доменные объекты;
- сам решать, можно ли заменять текущие данные.

## Что должен делать FileValidator

`FileValidator` должен проверять данные файла до загрузки в сервисы.

Минимальный набор проверок:

1. В файле есть секции:
   - `experiments`
   - `runs`
   - `runResults`

2. Внутри каждой секции `id` уникальны.

3. Обязательные поля не пустые.

4. Числовые поля корректны:
   - `id > 0`
   - `experimentId > 0`
   - `runId > 0`

5. Значения соответствуют правилам домена:
   - `Experiment.name` не пустое, длина до 128
   - `description` до 512
   - `ownerUsername` не пустой, длина до 128
   - `Run.name` не пустое, длина до 128
   - `operatorName` не пустой, длина до 64
   - `RunResult.unit` не пустой, длина до 16
   - `comment` до 128
   - `pH` в диапазоне 0..14
   - `Concentration >= 0`

6. Корректны enum-значения:
   - `MeasurementParam` должен парситься из допустимого набора

7. Корректны даты:
   - `createdAt` и `updatedAt` присутствуют
   - парсятся в `Instant`

8. Корректны связи:
   - каждый `Run.experimentId` ссылается на существующий `Experiment.id`
   - каждый `RunResult.runId` ссылается на существующий `Run.id`

Критично:

- валидатор должен кидать осмысленный `ValidationException`;
- он не должен падать `NullPointerException` на битом файле.

## Что должен делать DataManager

`DataManager` можно оставить как фасад между CLI и storage/service слоями, но собрать заново.

### save

План для `save`:

1. Взять данные из сервисов.
2. Сконвертировать доменные объекты в DTO-записи файла.
3. Собрать `DataSnapshot`.
4. Передать его в `FileStorage.save(...)`.

### load

План для `load`:

1. Прочитать `DataSnapshot` из файла через `FileStorage`.
2. Провалидировать snapshot через `FileValidator`.
3. На основе snapshot восстановить доменные объекты через restore/factory методы.
4. Загрузить их во временные сервисы.
5. Проверить, что временные сервисы полностью собраны.
6. Только после этого заменить данные в основных сервисах.

Это важный принцип: если файл битый, текущие данные в памяти не должны портиться.

## Что удалить или переписать из текущего stage 3

С высокой вероятностью:

1. Полностью переписать текущий `JsonFileStorage`.
   Причина: он завязан на прямую десериализацию домена.

2. Полностью переписать текущий `JsonDataValidator`.
   Причина: в нём уже есть полезный список проверок, но реализация сырая, местами небезопасная и привязана не к DTO, а к доменным объектам.

3. Полностью переписать `DataManager`.
   Причина: сама идея фасада нормальная, но внутренняя логика должна работать через DTO + validator + restore.

4. Вернуть или заново создать DTO:
   - `DataSnapshot`
   - `ExperimentRecord`
   - `RunRecord`
   - `RunResultRecord`

5. Удалить мусорные и временные файлы, если они попали в ветку.

## Предлагаемый порядок реализации

### Шаг 1. Зафиксировать restore-контракт домена

Добавить в доменные классы factory-методы или restore-конструкторы:

- `Experiment.restore(...)`
- `Run.restore(...)`
- `RunResult.restore(...)`

Они должны:

- принимать `id`, все поля и timestamps;
- валидировать данные;
- создавать корректный объект без генерации новых дат "на сейчас".

### Шаг 2. Сделать DTO для файла

Создать или вернуть file DTO:

- `DataSnapshot`
- `ExperimentRecord`
- `RunRecord`
- `RunResultRecord`

DTO должны быть простыми контейнерами данных, без бизнес-логики.

### Шаг 3. Сделать mapper между domain и DTO

Добавить класс или набор методов, который умеет:

- из домена делать record;
- из record делать домен через `restore(...)`.

Это уберет прямую связку `Jackson <-> domain`.

### Шаг 4. Сделать нормальный JsonFileStorage

Он должен работать только с `DataSnapshot`.

При записи:

- писать читаемый JSON;
- использовать UTF-8;
- корректно писать даты как строки.

При чтении:

- читать `DataSnapshot`;
- не пытаться создавать доменные объекты.

### Шаг 5. Сделать безопасный FileValidator

Валидировать именно DTO, а не доменные объекты.

Плюсы:

- можно поймать ошибки до восстановления домена;
- не будет зависимости от того, как Jackson умеет собирать сущности;
- сообщения об ошибках будут чище.

### Шаг 6. Пересобрать DataManager

Сделать новый сценарий:

- `save`: services -> mapper -> snapshot -> file
- `load`: file -> snapshot -> validator -> mapper/restore -> temp services -> replace

### Шаг 7. Подключить CLI

CLI должен только:

- вызвать `dataManager.saveToFile(path)` или `loadFromFile(path)`;
- вывести нормальное сообщение об успехе или ошибке.

CLI не должен заниматься логикой восстановления.

### Шаг 8. Закрыть тестами

Без этого stage 3 считать готовым нельзя.

## Какие тесты нужны обязательно

### Unit tests для FileValidator

Проверить:

- отсутствие обязательной секции;
- дубликаты `id`;
- битые ссылки;
- пустые обязательные поля;
- слишком длинные строки;
- невалидный `param`;
- невалидные даты;
- невалидные значения `pH` и `Concentration`.

### Unit tests для mapper/restore

Проверить:

- domain -> DTO
- DTO -> domain
- сохранение `id`
- сохранение `createdAt`
- сохранение `updatedAt`

### Integration tests для DataManager

Проверить:

1. round-trip:
   - создать данные в сервисах;
   - сохранить в файл;
   - загрузить из файла;
   - убедиться, что данные совпали

2. `nextId` после load:
   - после загрузки новый объект получает `maxId + 1`

3. защита текущих данных:
   - если файл битый, существующие данные в сервисах не затираются

4. пустой snapshot:
   - пустые коллекции корректно загружаются

### CLI tests

Проверить:

- `load <path>` на валидном JSON
- `load <path>` на битом JSON
- `save <path>`

## Что будет итогом готового этапа 3

Этап 3 можно считать сделанным, если выполняются все условия:

1. `save <path>` сохраняет полное состояние в JSON.
2. `load <path>` реально восстанавливает данные из этого JSON.
3. После `load` связи и timestamps не теряются.
4. Новые `id` после загрузки продолжают последовательность корректно.
5. Битый JSON отлавливается понятной ошибкой.
6. При неудачной загрузке текущие данные не повреждаются.
7. Есть тесты на round-trip и невалидные файлы.

## Короткий вывод

По работе:

- этапы 1 и 2 оставляем как базу;
- текущий stage 3 почти весь переписываем;
- домен и сервисы меняем минимально, только чтобы поддержать restore;
- файловый слой делаем заново и правильно;
- после этого уже можно спокойно идти в этап 4 с UI поверх стабильных сервисов.
