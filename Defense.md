## Архитектурная схема
- `Experiment` является корневой сущностью
- `Run` ссылается на `Experiment` через `experimentId`
- `RunResult` ссылается на `Run` через `runId`
- доменные классы не вызывают сервисы
- сервисы создают, ищут, обновляют и удаляют сущности
- сервисы проверяют межсущностные связи
- CLI не работает напрямую с коллекциями и не содержит бизнес-логику

Что говорить устно:

Связи между сущностями проверяются только в сервисах, потому что только сервисный слой знает про все коллекции. Доменный объект не должен ходить в сервисы и искать внешние сущности.

## Почему `id` генерируется в сервисах
- `id` относится к жизненному циклу коллекции, а не к внутреннему состоянию одной сущности
- доменная сущность получает `id` уже готовым через конструктор
- `ExperimentService`, `RunService` и `RunResultService` сами вызывают `IdGenerator.generateId()`

Короткий ответ:

`id` генерируется там, где живет коллекция объектов. Это ответственность сервисного слоя, а не доменной модели.

## Доменная модель

### `Experiment`
Поля:
- `id`
- `name`
- `description`
- `ownerUsername`
- `createdAt`
- `updatedAt`

Инварианты:
- `id > 0`
- `name` обязателен, не пустой, длина до 128
- `description` необязателен, длина до 512
- `ownerUsername` обязателен, не пустой, длина до 128

Изменение состояния:
- есть сеттеры
- есть атомарный `update(name, description, ownerUsername)`
- `updatedAt` обновляется при каждом изменении

### `Run`
Поля:
- `id`
- `experimentId`
- `name`
- `operatorName`
- `createdAt`
- `updatedAt`

Инварианты:
- `id > 0`
- `experimentId > 0`
- `name` обязателен, не пустой, длина до 128
- `operatorName` обязателен, не пустой, длина до 64

Изменение состояния:
- есть сеттеры
- есть атомарный `update(name, operatorName)`
- `updatedAt` обновляется при изменении

### `RunResult`
Поля:
- `id`
- `runId`
- `param`
- `value`
- `unit`
- `comment`
- `createdAt`
- `updatedAt`

Инварианты:
- `id > 0`
- `runId > 0`
- `param != null`
- `unit` обязателен, не пустой, длина до 16
- `comment` необязателен, длина до 128

Проверка `value` зависит от параметра:
- для `pH` допустим диапазон от 0 до 14
- для `Temperature` отрицательные значения разрешены
- для `Concentration` значение не может быть отрицательным

Важная деталь для защиты:

В `setParam(...)` сначала валидируется новый параметр вместе с текущим `value`, и только потом меняется поле. Это защищает объект от частичного повреждения состояния.

### `MeasurementParam`
Поддерживаемые значения:
- `pH`
- `Temperature`
- `Concentration`

## Почему в домене есть `update(...)`, а в сервисе тоже есть `update(...)`
- доменный `update(...)` меняет состояние одного уже найденного объекта
- сервисный `update(id, ...)` сначала находит объект в коллекции по `id`
- после этого сервис делегирует изменение доменному объекту

Короткий ответ:

Сервис отвечает за поиск объекта в коллекции, домен отвечает за корректное изменение собственного состояния.

## Почему обновления сделаны атомарно
- если из сервиса вызывать несколько сеттеров подряд, объект можно испортить частично
- поэтому сервис не меняет поля по одному
- вместо этого сервис вызывает один доменный `update(...)`
- внутри `update(...)` сначала валидируются все входные данные, потом уже применяется новое состояние

Это один из сильных аргументов на защите, потому что показывает, что проект не просто "работает", а архитектурно продуман.

## Сервисный слой

### `ExperimentService`
Реально реализовано:
- `add(name, description, ownerUsername)`
- `getById(id)`
- `list()`
- `update(id, name, description, ownerUsername)`
- `remove(id)`

Особенности:
- хранение в `TreeMap<Long, Experiment>`
- `list()` возвращает `List.copyOf(...)`
- при отсутствии объекта выбрасывается `ValidationException`

### `RunService`
Реально реализовано:
- `add(experimentId, name, operatorName)`
- `getById(id)`
- `list()`
- `listByExperimentId(experimentId)`
- `update(id, name, operatorName)`
- `remove(id)`

Особенности:
- до создания `Run` вызывается `experimentService.getById(experimentId)`
- это гарантирует, что прогон нельзя создать для несуществующего эксперимента

### `RunResultService`
Реально реализовано:
- `add(runId, param, value, unit, comment)`
- `getById(id)`
- `list()`
- `listByRunId(runId)`
- `update(id, param, value, unit, comment)`
- `remove(id)`

Особенности:
- до создания `RunResult` вызывается `runService.getById(runId)`
- это гарантирует, что результат нельзя создать для несуществующего прогона

Текущий список команд:
- `help`
- `exit`
- `exp_add`

Что важно проговорить:
- `Main` только делегирует запуск в `CliRunner.run()`
- `CliRunner` держит цикл чтения команд, `Scanner`, разбор аргументов и печать сообщений
- бизнес-логика остается в сервисах и доменных классах

## Как работает разбор команд
- строка делится на имя команды и строку аргументов через `split("\\s+", 2)`
- имя команды приводится к нижнему регистру
- дальше `switch` обрабатывает команду по имени
- ошибки валидации ловятся и показываются пользователю как `Validation error: ...`
- неожиданные runtime-ошибки тоже перехватываются, чтобы CLI не падал стек-трейсом

## Интерактивные команды
Команды с пошаговым вводом полей:
- `exp_add`
- `run_add`
- `res_add`

Как это устроено:
- `readRequiredValue(...)` не принимает пустую строку
- `readOptionalValue(...)` превращает пустой ввод в `null`
- `readRequiredDouble(...)` повторяет запрос, пока не введено число
- `readMeasurementParam(...)` повторяет запрос, пока не введен допустимый параметр

Важно:

Параметры измерения в CLI читаются без учета регистра. То есть `ph`, `PH` и `pH` будут распознаны как `MeasurementParam.pH`.

- за один вызов меняется только одно поле
- допустимые поля:

Что сказать устно:

Команда специально упрощена до обновления одного поля за вызов. Так проще и код, и контракт, и объяснение на защите.

### `res_add`
- принимает только `runId` в самой команде
- дальше интерактивно спрашивает:

- `Unit`

Важно:

### `res_list`
- базовый вариант: `res_list <runId>`
- с фильтром: `res_list <runId> --param PARAM`
- `PARAM` должен быть одним из:
- `pH`
- `Temperature`
- `Concentration`

Важно:

Фильтр по `--param` применяется на уровне CLI после получения списка результатов прогона из сервиса.

## Что делает `exp_summary`
- получает эксперимент по `id`
- получает все его прогоны через `runService.listByExperimentId(...)`
- для каждого прогона получает результаты через `runResultService.listByRunId(...)`
- группирует числовые значения по `MeasurementParam` в `EnumMap`
- для каждой группы считает `count`, `min`, `max`, `avg`
- выводит статистику в фиксированном формате

Короткий ответ:

`exp_summary` реализован простыми структурами данных и обычными циклами. Это осознанно простое и прозрачное решение, которое легко показать и объяснить на защите.

## Что могут спросить и как отвечать

### Почему доменные классы не знают про сервисы
Потому что доменная модель должна отвечать только за собственные инварианты и состояние. Если доменный класс начнет вызывать сервисы, слои смешаются.

### Почему связь `Run -> Experiment` проверяется в `RunService`
Потому что только сервис знает про коллекцию экспериментов. Сам объект `Run` может проверить только корректность `experimentId` как значения, но не существование внешнего объекта.

### Почему связь `RunResult -> Run` проверяется в `RunResultService`
По той же причине: это внешняя ссылка на другую коллекцию, и проверять ее должен сервисный слой.

### Почему `list()` возвращает копию
Чтобы внешний код не получил прямой доступ к внутренней коллекции сервиса и не менял ее в обход сервиса.

### Почему выбран `TreeMap`
Потому что он дает стабильное хранение по ключу `id` и предсказуемый порядок обхода по возрастанию ключей, что удобно для CLI и тестов.

### Почему `Main` такой маленький
Потому что точка входа должна оставаться максимально тонкой. Вся CLI-инфраструктура вынесена в `CliRunner`.
