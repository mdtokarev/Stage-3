# Stage 4 Plan

## Цель этапа 4

Сделать JavaFX UI поверх уже готовых слоев `domain`, `service`, `validation`, `storage`, не перенося бизнес-логику в интерфейс и не ломая сценарии этапов 1-3.

Готовый этап 4 должен дать пользователю графический интерфейс для:

1. Просмотра списка экспериментов.
2. Просмотра прогонов выбранного эксперимента.
3. Просмотра результатов выбранного прогона.
4. Создания и обновления сущностей через формы.
5. Вызова уже готовых сценариев `save/load`.
6. Просмотра ошибок в понятном виде, без падения приложения.

Главный принцип: JavaFX UI это новый внешний слой над уже существующими сервисами.  
Сервисы и доменные классы остаются источником бизнес-правил, а UI только собирает ввод, вызывает сервисы и показывает результат.

## Что не надо делать

Не надо:

- переносить валидацию из домена и сервисов в контроллеры UI;
- дублировать логику `DataManager`, `ExperimentService`, `RunService`, `RunResultService` внутри JavaFX;
- связывать элементы интерфейса напрямую с файловым JSON-слоем;
- выбрасывать CLI, если он уже работает: UI должен жить рядом, а не вместо архитектуры;
- начинать с красивых деталей раньше, чем появится рабочий поток `experiment -> run -> result`.

## Целевая архитектура этапа 4

Рекомендуемый набор пакетов:

- `ui`
  JavaFX entry point и сборка приложения.
- `ui.view`
  Построение главного окна, `TableView`, toolbar, menu и простых layout-контейнеров.
- `ui.controller`
  Контроллеры экранов и обработчики действий пользователя.
- `ui.viewmodel`
  Простые модели для отображения в `TableView` и деталях, если код начнет разрастаться.
- `ui.dialog`
  Формы создания/редактирования и окна ошибок.
- `ui.mapper`
  Преобразование `domain -> UI model`, если начнет разрастаться код таблиц.

Слои взаимодействуют так:

`JavaFX view/controller -> service/DataManager -> domain/storage`

Важно сохранить правило:

- `domain` ничего не знает про UI;
- `service` ничего не знает про JavaFX;
- `storage` ничего не знает про сцены, таблицы и кнопки.

Для варианта с "таблицами" базовым UI-виджетом должен быть именно `TableView`, а не произвольные списки строк.

## Минимальный функциональный объем

Для первого рабочего варианта UI достаточно закрыть такой набор:

1. Главное окно.
   В нем должны быть:
   - `TableView` экспериментов;
   - `TableView` прогонов выбранного эксперимента;
   - `TableView` результатов выбранного прогона;
   - зона деталей или краткой информации по выбранной сущности.

2. Действия с экспериментами:
   - создать;
   - обновить;
   - показать summary;
   - удалить при необходимости, если это уже поддерживается сервисом.

3. Действия с прогонами:
   - создать для выбранного эксперимента;
   - показать детали;
   - удалить при необходимости, если это уже поддерживается сервисом.

4. Действия с результатами прогонов:
   - создать для выбранного прогона;
   - фильтровать по `MeasurementParam`;
   - удалить при необходимости, если это уже поддерживается сервисом.

5. Файловые операции:
   - `Save`;
   - `Load`.

6. Обработка ошибок:
   - ошибки валидации;
   - ошибки загрузки файла;
   - ошибки пустого выбора;
   - ошибки некорректного ввода чисел.

## Минимальные колонки таблиц

Чтобы не расползаться по UI на первом проходе, достаточно такого набора колонок:

- `ExperimentTableView`
  - `id`
  - `name`
  - `ownerUsername`
  - `createdAt`

- `RunTableView`
  - `id`
  - `name`
  - `operatorName`
  - `createdAt`

- `RunResultTableView`
  - `id`
  - `param`
  - `value`
  - `unit`
  - `comment`

Если понадобится, даты можно сначала показывать строкой без сложного форматирования, а полировку оставить на конец.

## Предлагаемый UI-сценарий

Самый безопасный и понятный вариант для учебного проекта:

1. Слева таблица экспериментов.
2. По центру таблица прогонов выбранного эксперимента.
3. Справа таблица результатов выбранного прогона.
4. Сверху toolbar или menu bar:
   - `New Experiment`
   - `Save`
   - `Load`
   - `Refresh`
5. Снизу или справа панель деталей и summary.

Такой layout хорошо ложится на предметную область и сразу показывает связи:

- один `Experiment` -> много `Run`;
- один `Run` -> много `RunResult`.

Это и есть рекомендуемый `master-detail` для вашего проекта:

- master 1: таблица `Experiment`
- detail 1: таблица `Run` выбранного `Experiment`
- detail 2: таблица `RunResult` выбранного `Run`

## Технический план реализации

### Шаг 1. Подготовить Gradle под JavaFX

Нужно:

- добавить зависимости JavaFX;
- выбрать способ запуска JavaFX через Gradle;
- оставить существующий `cli.Main`, а для UI добавить отдельную точку входа, например `ui.UiMain`.

Практически лучше иметь два entry point:

- `cli.Main` для консоли;
- `ui.UiMain` для JavaFX.

Если понадобится, можно добавить отдельную Gradle-задачу вроде `runUi`.

### Шаг 2. Сделать JavaFX entry point

Нужен класс на базе `Application`, который:

- создает общие экземпляры `ExperimentService`, `RunService`, `RunResultService`, `DataManager`;
- поднимает главное окно;
- передает сервисы в контроллеры или в корневой coordinator-класс.

На этом шаге не надо сразу делать много экранов.  
Достаточно открыть пустое рабочее окно и убедиться, что JavaFX стартует.

### Шаг 3. Собрать главный экран

На главном экране сначала реализовать:

- таблицу экспериментов;
- таблицу прогонов;
- таблицу результатов;
- кнопки основных действий.

Пока без сложных диалогов можно даже временно отключить часть кнопок, если они не подключены.

Главная задача шага: добиться корректной цепочки обновления данных:

1. выбрали experiment;
2. автоматически подгрузили его runs;
3. выбрали run;
4. автоматически подгрузили его runResults.

На этом же шаге полезно зафиксировать базовое правило selection:

- если `Experiment` не выбран, таблица `Run` пустая;
- если `Run` не выбран, таблица `RunResult` пустая;
- при смене `Experiment` сбрасывается selection у `Run` и `RunResult`;
- при смене `Run` сбрасывается старый detail, если он уже невалиден.

### Шаг 4. Подключить создание эксперимента

Сделать диалог или форму для `Experiment`:

- `name`
- `description`
- `ownerUsername`

После успешного создания:

- обновить список экспериментов;
- выделить новый объект;
- показать сообщение об успехе по желанию.

Если сервис или домен кидает `ValidationException`, UI должен показать человеку понятный alert.

### Шаг 5. Подключить создание run

Сделать форму создания `Run` для уже выбранного `Experiment`.

Поля:

- `name`
- `operatorName`

Важно:

- без выбранного эксперимента форма не должна создавать `Run`;
- сообщение об ошибке должно быть пользовательским, а не стек-трейсом.

### Шаг 6. Подключить создание run result

Сделать форму создания `RunResult` для выбранного `Run`.

Поля:

- `param`
- `value`
- `unit`
- `comment`

Особенно аккуратно обработать:

- выбор `MeasurementParam` через `ComboBox`;
- ввод `value` как числа;
- доменные ограничения вроде диапазона `pH` и `Concentration >= 0`.

### Шаг 7. Подключить редактирование

После создания основных форм добавить редактирование:

- `Experiment.update(...)`
- те обновления, которые уже поддержаны доменными и сервисными методами

Лучше идти от уже реализованного контракта сервисов, а не выдумывать новый "UI-only" сценарий изменения данных.

Если обновление `Run` и `RunResult` пока не реализовано на сервисном уровне, не надо пытаться спрятать это в UI.  
Сначала либо оставить только просмотр/создание, либо отдельно расширить сервисный слой.

То же правило относится к удалению:

- если delete уже есть в сервисе, UI может его вызывать;
- если delete еще не оформлен как нормальный сервисный сценарий, в первой версии UI его можно не включать.

### Шаг 8. Подключить summary эксперимента

Так как в CLI уже есть `exp_summary`, для UI логично сделать:

- кнопку `Show Summary`;
- отдельный dialog или pane с агрегатами `count/min/max/avg` по каждому `MeasurementParam`.

Расчет summary должен переиспользовать существующую логику, а не дублироваться в UI.

Если summary сейчас реализован только как кусок CLI-логики, лучше вынести его в отдельный сервисный метод перед подключением UI.

Это желательно считать не опцией, а обязательной подготовительной задачей перед полноценным UI-summary:

- CLI не должен оставаться единственным местом, где живет расчет summary;
- UI должен вызывать готовый сервисный метод и только отображать результат.

### Шаг 9. Подключить Save/Load

Добавить действия меню или toolbar:

- `Save`
- `Load`

Через стандартный file chooser:

- выбрать путь;
- вызвать `dataManager.save(...)` или `load(...)`;
- показать сообщение об успехе или ошибке.

Критично:

- при ошибке `load` UI не должен портить текущие отображаемые данные;
- после успешного `load` интерфейс должен полностью перерисовать списки и сбросить старые selection, если они уже невалидны;
- после `create` полезно автоматически выделять новый объект;
- после `delete` нужно очищать зависимые таблицы и детали, если выбор стал невалидным.

### Шаг 10. Полировка UX

После того как все работает, довести удобство:

- блокировать кнопки, если нет выбранной сущности;
- добавлять confirm dialog перед удалением, если удаление есть;
- делать понятные заголовки ошибок;
- добавить пустые состояния таблиц;
- аккуратно форматировать даты и числа в таблицах.

## Что лучше вынести из CLI перед UI

Перед стартом полноценного JavaFX-этапа полезно проверить, не осталось ли логики, слишком привязанной к CLI.

Особенно стоит вынести в сервисный слой, если еще не вынесено:

- summary эксперимента;
- формат удобного представления данных для деталей сущности;
- общие операции refresh/list/filter, чтобы UI не вычислял предметную логику сам.

Идея простая: UI не должен знать, как считать summary или как проверять допустимость связи.  
UI должен только вызывать готовые методы.

## Тестовый план

Для учебного проекта не обязательно делать тяжелые UI-тесты на все, но минимум нужен.

### Unit tests

Покрыть:

- вспомогательные UI-mapper классы, если они появятся;
- парсинг/преобразование полей формы;
- логику summary, если ее придется выносить в сервис.

### Integration tests

Проверить сценарии:

1. создание `Experiment` через UI-обработчик;
2. создание `Run` только при выбранном `Experiment`;
3. создание `RunResult` только при выбранном `Run`;
4. refresh таблиц после `save/load`;
5. корректное отображение ошибок валидации.

Если полноценные JavaFX-тесты окажутся слишком тяжелыми, допустимый компромисс:

- максимум логики вынести из controller в обычные классы;
- тестировать эти классы отдельно;
- вручную проверить экраны.

### Ручная проверка

Обязательный smoke test:

1. запустить UI;
2. создать эксперимент;
3. создать прогон;
4. создать несколько результатов;
5. проверить summary;
6. сохранить в файл;
7. перезапустить приложение;
8. загрузить файл;
9. убедиться, что данные, связи и `id` продолжаются корректно.

Этот список можно использовать и как минимальный сценарий ручной демонстрации на защите.

## Основные риски этапа 4

1. Смешивание UI-логики и бизнес-логики.
   Это главный риск. Если контроллеры начнут сами проверять связи и считать summary, архитектура быстро поплывет.

2. Слишком ранний уход в FXML и сложную структуру экранов.
   Если сначала строить много экранов, можно увязнуть. Лучше сначала один рабочий главный экран.

3. Дублирование контрактов CLI.
   UI не должен изобретать отдельные правила обновления или отдельный формат ошибок.

4. Неполная синхронизация после `load`.
   После загрузки надо обновлять все списки и выделения, иначе UI покажет старое состояние.

5. Проблемы с JavaFX-конфигурацией в Gradle.
   Это лучше закрыть самым первым техническим шагом.

## Рекомендуемый порядок работ

Оптимальный порядок такой:

1. Настроить JavaFX в `build.gradle`.
2. Запустить пустое JavaFX-окно.
3. Сделать главное окно с тремя таблицами.
4. Подключить список и выбор `Experiment`.
5. Подключить список и выбор `Run`.
6. Подключить список и выбор `RunResult`.
7. Подключить создание `Experiment`.
8. Подключить создание `Run`.
9. Подключить создание `RunResult`.
10. Вынести и подключить summary через сервисный слой.
11. Подключить `save/load`.
12. Довести UX и проверки.
13. Обновить `README.md` под запуск UI.

## Критерии готовности этапа 4

Этап 4 можно считать готовым, если:

1. JavaFX-приложение стабильно запускается через Gradle.
2. UI работает поверх тех же сервисов, что и CLI.
3. Через UI можно создать `Experiment`, `Run`, `RunResult`.
4. Видны связи между сущностями через выбор в таблицах.
5. Ошибки валидации показываются человеку понятно.
6. `save/load` работают из UI и не ломают текущие данные.
7. После `load` продолжается корректная генерация `id`.
8. Архитектура проекта по-прежнему разделена на `domain`, `service`, `validation`, `cli`, `storage`, `ui`.

## Короткий вывод

Stage 4 лучше делать как тонкий JavaFX-слой над уже готовым ядром.  
Самая правильная стратегия: сначала поднять минимально рабочее окно с тремя связанными таблицами, затем подключать формы создания, потом `summary`, потом `save/load`, и только после этого заниматься полировкой интерфейса.
